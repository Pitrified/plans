# lfs issues

## problem

i have a github pat with

```
Repository access
This token has access to all repositories owned by you.

User permissions
This token does not have any user permissions.

Repository permissions
Read access to metadata
Read and Write access to actions, code, pages, pull requests, and workflows
```

when pushing with it

pmn@pmn-14G4:~/bootstrap (master)$ git push -v
Pushing to https://github.com/Pitrified/bootstrap.git
error: Authentication error: Authentication required: You must have push access to verify locks
error: failed to push some refs to 'https://github.com/Pitrified/bootstrap.git'

which permission should the token have?

why is lfs kicking in even if the repository doesn't have any LFS files?

## analysis

The install script ran `git lfs install` from inside `~/bootstrap`, so git-lfs
installed a per-repo `pre-push` hook (`.git/hooks/pre-push`). That hook runs
`git lfs pre-push` on every push, which makes an LFS lock-verification API call
*before* checking whether the repo has any LFS files. GitHub returns 403 for
that call with a fine-grained PAT, and git-lfs aborts the push with
"Authentication required: You must have push access to verify locks".

- The push failure is unrelated to LFS files - there are none (`git lfs ls-files`
  is empty). It is purely the lock-verify probe in the hook.
- The token already has the permission needed to push code
  (Repository -> Contents = "code" Read and Write). The failing call is the
  separate LFS *locks* API, which fine-grained PATs do not satisfy reliably.
  This is a known GitHub limitation, not a missing scope. A classic PAT with
  the `repo` scope would satisfy it, but we don't use locking so it isn't worth it.

## solution

Scoped to bootstrap only - other repos (e.g. lang-tools) genuinely use LFS and
must not be touched.

Applied on this box:
- `git lfs uninstall --local` inside ~/bootstrap - removed the stray pre-push hook
  and local lfs config. bootstrap has no LFS files, so it should never have had them.
  Global filters and every other repo are untouched. Push is now plain git, no
  lock-verify step at all.
- Reverted the earlier global `lfs.locksverify false` (it was leaking into LFS
  repos via inheritance). Global config is back to its original state.

Script fix (`~/bootstrap/install_git_lfs.sh`) so it doesn't recur on new machines:
- `git lfs install --skip-repo` instead of `git lfs install` - sets up the global
  filters without dropping a pre-push hook into whatever repo is the CWD.
  Real LFS repos still work: global filters are all they need, and their per-repo
  hooks are installed when cloned.

Note on LFS repos + fine-grained PAT: if an actual LFS repo ever hits the same
403 on push, fix it per-repo (`git -C <repo> config lfs.locksverify false`),
which only disables the file-locking check for that repo - do NOT set it globally.
